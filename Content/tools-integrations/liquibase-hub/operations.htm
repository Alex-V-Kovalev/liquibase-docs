Hub Operations

Operations are the records of and activity, such as the various updates, rollbacks, changelogSync, and dropAll commands reported to a Hub Project by a registered . These operations are the primary data from which Hub derives values.

Also, an operation is an action taken by CLI on a source database. Hub allows you to track when the operation starts, what it intends to do, its progress, and when it finishes if everything goes as expected.

All operations are related to an Organization and are performed in the context of a Project. Operations cannot be transferred from one Organization or Project to another.

In most cases, operations relate to changes performed on the database:

  • update – new changes are added.
  • rollback – new changes are added, reversing previous changes.
  • changelogSync – new changes are added but not deployed by .
  • dropAll – new changes are added, dropping database objects owned by the user.

You can set the level of data, which you want to send to Hub, in your file:

liquibase.hub.mode:[all|meta|off]
  • all sends all data, including the body, generated SQL, and logs (the default value is all).
  • meta does not send the body, generated SQL, or operation-event and change-event logs. This mode sends the name, author, id, timestamps, version, and others.
  • off sends no data to Hub.

Alternatively, you can set the level of data specifying SET JAVA_OPTS="-DLiquibaseHubMode=[all|meta|off]" (the default value is all) for Windows, Linux, or MacOS.

Hub commands

Hub tracks the following commands:

  • update
  • updateCount
  • updateToTag
  • rollback
  • rollbackToDate
  • rollbackCount
  • rollbackOneChangeSet
  • rollbackOneUpdate
  • changelogSync
  • dropAll

Each command automatically synchronizes the local table with the Hub’s database and sends information to a Hub Project. If the Internet connection is lost or Hub doesn’t respond, will still execute the commands and will not be blocked by a lack of the Hub communication.

Also, there are two new commands specific to Hub functionality:

  • registerChangeLog
  • syncHub

For more information, see Using commands in Hub.

Reports

Hub provides reports based on your and activity. You can filter your Operation reports by Project, connections, operation type, date, result, , or users.

The following reports are available:

  • update
  • rollback
  • changelogSync
  • dropAll (coming soon)

The update Operation Report

The report based on the update operations is used to track s that were deployed, their execution status, timing, and various client and runtime parameters.

Hub functionality includes the following:

  • Operation summary, which shows:
    • Connection information with the Hub name of the connection, URL, user, and version of the database.
    • Client information with the version, hostname, and interface, such as CLI, Maven (and others coming soon).
    • Operation parameters such as labels, contexts, tracking table schema, tracking table catalog.
    • Runtime information with a start time, end time, and total time of the deployment.
  • overview, which represents the that was a part of your operation, its author, filepath, comment associated with that , and outcome. The outcome of the execution is shown as Pending, Success, or Fail.
  • details, which include a title, id, author, filepath, and duration of running an individual . details are available if hub.mode=all:
    • body – the actual data that was passed to .
    • Generated SQL that was executed by – the representation of the that was run on the database.
    • Execution output – the log messages associated with the individual s.
    • Additional attributes – the attributes that are specified in the .

The Hub Project associated with the is updated both with a failed or passed operations for each . It optionally includes the generated SQL executed by update.

The rollback Operation Report

The report based on the rollback operations presents the same details as the update operation report. However, it differs in the Operation Summary and Details.

The Operation Summary includes:

  • Operation parameters – the exact rollback command that was run—rollbackOneUpdate, rollbackToDate, rollbackCount, or rollbackOneChangeSet.
  • Command parameters – parameter added to the rollback commands. For example, the update ID that you need to pass to to run the rolbackOneUpdate command.

The Details field shows whether the rollback is automatically generated or with the user details provided. For example, if the rollback source is User Defined, the can include the following rollback information:

<!-- Rename table on rollback to preserve data -->
    <rollback>
       <renameTable oldTableName=”Favorites”
        newTableName=”ROLLBACK_FAVORITES”/>
    </rollback>

The Hub Project associated with the is updated both with a failed or passed rollback operations for each . It optionally includes the generated SQL executed by rollback.

The changelogSync Operation Report

The report based on the changelogSync operation presents only the meta information. It doesn’t include details or the generated SQL because the changelogSync command doesn’t make any changes to the database.

The changelogSync log output provides information about the s status.
