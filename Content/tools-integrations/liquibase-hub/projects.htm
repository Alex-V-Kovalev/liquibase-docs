Projects

A Project is a structured collection of the resources used by or 4.1+. An Organization can contain many Projects that include a list of your Targets, s with their s, and the Operations, which were run on the Project's database.

A or app user with a valid API Key can register a to a Project. Registering a enables the operations, such as update, rollback, changelogSync, and dropAll to be recorded and available for review as Operations reports in . For more information, see [%=General.Liquibase%] Hub Operations.

All members of an Organization can create Projects, register their to the Organization's Project, and send and view Operation data to these Projects. An Organization Admin can manage access to Projects in the Organization by adding or removing members from the Organization.

You can create Projects from the CLI during the registerChangeLog command process.

Targets

A target represents a database or schema where changes are made by using the operations that are tracked by —update, rollback, changelogSync, and dropAll.

Targets include your JDBC URL, target ID, the last operation with its date, time, and the that was used for the Operation. You can see the Operations related to targets in more detail by selecting the specific Target.

Additionally, the Targets view, which is similar to a single node in a pipeline, provides s that have been deployed to a Target. The targets listed have Status, and when you select it, you will be directed to the s page, which shows all s that have successfully been deployed to a specific Target.

Database schema changes to a specific Target are stored in the local tracking table that exists in that Target, which is synchronized with its counterpart when working with a registered . You cannot transfer Target's data between Projects or Organizations as it is automatically captured when you report commands to .

A Project can have multiple Targets. However, two Organizations and Projects should not share the same local table as this is the core element tracked by local usage, which is synchronized and reported on at . Therefore, it is a best practice to register the s to the same Project if your workflow shares the same table.
