Projects

A Hub Project is a structured collection of the resources used by or 4.1+. An Organization can contain many Projects that include a list of your Connections, Changelogs with their s, and the Operations, which were run on the Project's database.

A or app user with a valid Hub API Key can register a to a Project. Registering a enables the operations, such as update, rollback, changelogSync, and dropAll to be recorded and available for review as Operations reports in Hub. For more information, see Operations.

All members of an Organization can create Projects, register their to the Organization's Project, and send and view Operation data to these Projects. An Organization Admin can manage access to Projects in the Organization by adding or removing members from the Organization.

You can create Hub Projects from the CLI during the registerChangeLog command process.

Connections

A connection represents a database or schema where changes are made by using the operations that are tracked by Hub—update, rollback, changelogSync, and dropAll.

Connections include your JDBC URL, username, and the last operation with its date, time, and the that was used for the Operation. You can also see the Operations related to the connections in more detail by selecting the specific connection.

Database schema changes to a specific connection are stored in the local tracking table that exists in that connection, which is synchronized with its Hub counterpart when working with a registered . You cannot transfer Connections' data between Projects or Organizations as it is automatically captured when you report commands to Hub.

A Project can have multiple connections. However, two Organizations and Projects should not share the same local table as this is the core element tracked by local usage, which is synchronized and reported on at Hub. Therefore, it is a best practice to register the s to the same Project if your workflow shares the same table.

Changelogs

Hub relies on a unique identifier in your , called changeLogId, to synchronize the local deployments to your Hub Projects. Run the registerChangeLog command using or 4.1+ to connect an XML or formatted SQL to an existing or new Hub Project. (Additional formats coming soon!)

If you use nested s, register only the master without registering its nested s. For more information about nested s, see Liquibase Best Practices.

Once you have run the registerChangeLog command, it is a best practice to commit the to source control and notify others who use this to pull in changes, so that the same cannot be registered with Hub multiple times.
