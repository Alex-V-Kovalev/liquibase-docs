Maven generateChangeLog

generateChangeLog creates a file that has a sequence of s which describe how to re-create the current state of the database.

Uses

generateChangeLog is typically used when you want to capture the current state of a database, then apply those changes to any number of databases. This is typically only done when a project has an existing database but hasn't used before. See How to Set up with an Existing Project and Multiple Environments for more details.

When using the update goal to apply the changes in the , will not create a new database or schema. You must create them before applying the to it.

Maven configuration

Maven can be configured in multiple ways. One way is to define your configuration properties in your pom.xml file. To configure your pom.xml file, refer to Configuring s in your Maven POM File. If your pom.xml does not reference a file, include <changeLogFile> specify your changelog file here</changeLogFile> to the pom.xml.

Also, before running the generateChangeLog goal, add <outputChangeLogFile>specify the file name you want to generate here</outputChangeLogFile> to your pom.xml file.

Running the generateChangeLog Maven goal

Running the generateChangeLog goal requires a Maven project to be implemented.

To run the goal, type the following in your command prompt:

mvn liquibase:generateChangeLog
Output

The generateChangeLog goal generates a that contains all your objects (represented as s) and places the file in the same directory where the goal was ran.

The extension provided determines the format of the , so if you specify the filename as changelog.xml you will get an XML formatted . However, if you specify the filename as changelog.yaml or changelog.json or changelog.postgresql.sql you will get s formatted in YAML or JSON or SQL, respectively.

When generating an SQL formatted , you must specify the short name of the targeted database type as part of the filename (as shown in the previous sentence). To find or check your database’s short name, see Supported Databases.

produces a similar file to the one that is produced by ; however, it also produces the additional detail and files and supports specific stored logic information. The -supported stored logic types are:

  • package
  • packagebody
  • function
  • stored procedure
  • trigger
  • view

Not all database platforms support all these types.

If you choose a SQL formatted , the information about these stored logic types will be placed directly in the file.

If you choose an XML, YAML, or JSON format, the information about these types will be stored in a directory called Objects, which will be placed at the same level as your . The Objects directory contains a subdirectory for each of the listed stored logic types that contain the definition information for any items of that type found in the database. The generated file will contain references to that definition information. The following example illustrates an XML formatted showing such references:

  xmlns="http://www.liquibase.org/xml/ns/dbchangelog"

  xmlns:pro="http://www.liquibase.org/xml/ns/pro"
  xsi:schemaLocation="http://www.liquibase.org/xml/ns/dbchangelog
  http://www.liquibase.org/xml/ns/dbchangelog/dbchangelog-4.1.1.xsd">
	<changeSet  author="Administrator (generated)"  id="1571345362466-8">
	   <pro:createTrigger  disabled="false"  path="objects/trigger/TS_T_EXEMPLAR_SEQEXEMPLAR.sql"  relativeToChangelogFile="true"  tableName="T_EXEMPLAR"  triggerName="TS_T_EXEMPLAR_SEQEXEMPLAR"/>
	</changeSet>
	<changeSet  author="Administrator (generated)"  id="1571345362466-12">
	   <createView  fullDefinition="true"  path="objects/view/OREDERS_VIEW.sql"  relativeToChangelogFile="true"  viewName="OREDERS_VIEW"/>
	</changeSet>

Definition
liquibase.changeSetAuthor Specifies the author for s in the generated .

liquibase.diffTypes

Specifies the list of diff types to include in the expressed as a comma-separated list from:

liquibase.verbose [boolean] Controls the amount of output detail when you call the plugin. Default value is: false.
