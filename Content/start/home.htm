How to choose a format

uses two models for s:

  • The SQL model (.sql files): Write SQL statements directly in s in your , formatted with SQL comments.
  • The platform-agnostic model (.xml, .json, and .yaml files): Use  [%=General.changetypes%]s corresponding to SQL changes in each . then generates database-specific SQL for you. You can also insert raw SQL statements in your s.

You can mix and match one or more formats with simultaneously!

Here are some qualities to consider when choosing a format for :

  • Familiarity: If you already know the syntax for one of the formats, you can easily use it with .
  • Appearance: XML syntax is specific but highly verbose. If you want a less verbose format, try SQL, YAML, or JSON.
  • Tools: If your existing tools and processes are built around a particular format, such as SQL, you can easily integrate s of that format
  • Specificity: Formatted SQL s let you write specific SQL statements and control exactly how the database is modified. In contrast, XML, YAML, and JSON also support raw SQL statements, but are primarily meant to describe what kind of change you want to make:  handles the underlying database logic for you. XML, YAML, and JSON s can be particularly useful if you're working with multiple databases and don't want to write database-specific SQL, or if you want to generate automatic rollback statements for some changes.
