Transitioning from Datical DB to Datical Service

Comparison

For current users of Datical DB, the following list of differences can help orient you to Datical Service. 

Installation Location

  • Datical DB: Installed on a host for every user.
  • Datical Service: Installed on a server. Users access it through a browser. 

New Names for Familiar Components

  • Datical DB: CLI, GUI
  • Datical Service:
    • The Datical DB CLI is used as an Automation Client in automation solutions (ARA)
    • The Datical DB GUI is also referred to as a Desktop Client to distinguish it from Datical Service, which uses a server-based client (Deployment Management Console, or DMC). 

Database Access Requirements from Datical components

  • Datical DB: Requires network access to managed databases from every user installation. The database service user set up for Datical use must have permissions on each database. 
  • Datical Service: Requires network access to managed databases from Datical Service host. The database service user set up for Datical use must have permissions on each database. 

User Access to Datical

  • Datical DB: User has access to all (licensed) operations (package, forecast, deploy, rollback, others).
  • Datical Service: Logged-in users have role-based access to operations. Datical Server enforces authentication, which is integrated with customer-provided authentication service (Active Directory/LDAP). 

Packaging SQL Code

Get SQL code from an SCM repository, package it into changesets, and deploy it to REF database in a pipeline. 

  • Datical DB: User runs from automation, CLI, or GUI. 
  • Datical Service: User runs using Datical DB from automation or CLI.  Use additional options on the hammer command to specify Datical Service login credentials and specify the project. 

Operations/User Commands

  • Datical DB: User runs on installation host. Automation software runs them (requires Datical DB installation).
  • Datical Service: Requires Datical Service credentials. User runs on installation host. Automation software runs them (requires Datical DB installation).

Artifact Support 

Datical projects can be managed as artifacts. 

  • Datical DB: Artifacts are managed externally in an artifact repository, typically as part of operations run in automation software.
  • Datical Service: Artifacts are managed externally in an artifact repository, typically as part of operations run in automation software. CLI commands updated to support artifacts: immutable option in forecast and deploy commands). 

Automation

Use automation software to run Datical operations (package, forecast, deploy, commands). 

  • Datical DB: Automation software runs Datical DB CLI commands for operations (package, forecast, deploy)
  • Datical Service: Automation software runs Datical DB CLI commands for operations (package, forecast, deploy).  

Moving from Datical DB to Datical Service

This is a high-level list of things to do. 

Plan Users and Groups

  1. Understand Datical roles
  2. Reconcile Datical roles with organizational roles - who should have what permissions?
  3. Map to desired roles

Install Datical Service

Decisions: location in network (needs access SCM repository, databases)

Takeaway: URL to the service - give to users. 

Implement Roles for Users and Groups

If roles are implemented through groups, then most work is done when the group is imported. Need to review what Datical roles are assigned to each organizational group. 

Distribute URL to Users

Test logins and access.
