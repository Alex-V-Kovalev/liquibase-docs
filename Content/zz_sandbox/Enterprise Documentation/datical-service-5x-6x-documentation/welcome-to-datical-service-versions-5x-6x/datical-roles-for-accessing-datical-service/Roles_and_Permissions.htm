Assigning Roles to Groups

Typically you assign roles to groups according to their role in your organization. 

Groups are available in Datical Service from integration with your Auth service (AD/LDAP).  

You can also create groups or assign roles directly to users. 

Roles and the Release Pipeline in an Organization

In a large organization, responsibilities for creating and deploying database changes are distributed in different organizations. 

  • Developers write SQL code and check it in. 
    • Project Viewer role is appropriate. 
    • Step Deployer 
  • Database Analysts maintain database standards and practices. They check the code for compliance and move changes through the pipeline (deploy)
  • DevOps Staff - monitor changes through the release pipeline. 
    • If responsibility is assigned for the whole pipeline, assign Project Deployer role
    • If responsibility is assigned by environment (DEV, QA, PROD), assign Step Deployer role per step in the pipeline. 

Datical Role Distribution and Delegation

Datical roles are designed to map cleanly to organizational roles. 

System Administrator

The System Administrator role is for maintaining the system (Users, Groups, System Settings).

Users with System Administrator role can update their own roles. 

During installation you assign an initial user. It is assigned System Administrator role automatically. This user can bootstrap Datical Service by adding roles for projects to users and groups :  Project Creator, Project Administrator, Project Deployer. 

Create Project Role

This role is required to create or import projects. Organizationally it is related to the System Administrator role. 

The user who imports or creates a project is initially given Project Administrator role. During import/creation this user can assign a Project Administrator role to another user and remove himself. 

Project Roles

Each project is managed by a user with Project Administrator role. 

  • Project Administrator
    • Lead role
    • Add and remove users who can work with the project
    • Edit the project
  • Project Deployer - can deploy changes through the project. This is typically an operations or DevOps role, though it may also be a lead DB Administrator.
  • Project Analyst - can view and forecast changes but not deploy them. This can be a quality analyst or DB Administrator.
  • Project Viewer - helpful for developers who want to check the deployment status of changes they checked in. 

Step Roles

When organizational roles are segregated by release stage (DEV, QA, PROD), you can use Step roles. The Project Administrator assigns them. 

  • Step Deployer - can deploy this step, but not others in the release pipeline. This is typically an operations or DevOps role, though it may also be a lead DB Administrator.
  • Step Analyst - can view and forecast (but not deploy).This can be a quality analyst or DB Administrator.

Note

Step roles include Project Viewer role. Project Viewer contains permissions to view the project that contains the step, including project reports (forecast, deploy, packaging).
