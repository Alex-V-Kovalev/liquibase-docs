Configuring Quality Checks

provides default checks with default settings defined in the checks settings file.

Quality checks are either static or dynamic. A dynamic check can have multiple configurations, whereas a static check can have one.

  • A check is static if the attributes of the check cannot be customized. You can enable or disable static checks. You cannot copy, customize, or delete static checks.
  • A check is dynamic if there are settings you can customize. Dynamic checks contain values possible to modify according to your needs.
    • You can copy, customize, and reset dynamic checks but cannot delete them.
    • You can copy, customize, delete, and reset the copy of a dynamic check to the parent check’s default settings.
  • All checks can be enabled or disabled. users can run unlimited enabled checks, and users can run the first 5 enabled checks.

For now, provides the following static and dynamic checks:

Static checks

Warn on detection of GRANT statements

Value
Short name SqlGrantWarn
Description

The check warns when generated or raw SQL contains GRANT statements so that you can ensure that the privilege being granted will not lead to security issues.

For example, you have a formatted SQL file or a JSON, YAML, or XML file with at least one that includes the sql, sqlFile, or pro:sqlplus , and there is a statement that contains GRANT <privilege name>. When you execute liquibase checks run against that , you will receive an alert message about the with privileges.
Enabled by default Yes

Warn when a table drops

Value
Short name ChangeDropTableWarn
Description

The check warns when a table is being dropped so that you can ensure that dropping the table will not lead to unintentional loss of data. For example, you have one of the following:

  • A JSON, YAML, or XML file that includes the sql, sqlFile, or pro:sqlplus , and there is the statement that contains DROP TABLE

When you execute liquibase checks run against that , you will receive an alert message about the with a table being dropped.

Dynamic checks

Warn when SQL contains a specific pattern

Value
Short name SqlUserDefinedPatternCheck
Description

The check scans generated or raw SQL for the presence of specific patterns so that you can prevent security and velocity issues as early as possible.

Uses

You may have a JSON, YAML, or XML file with one or more that contain raw SQL or produce generated SQL that matches the regular expression. When you execute liquibase checks run against that , generates or accesses the SQL generated for the and checks for a match against the defined regular expression. After this, you will receive an alert message about the affected.

You may also have a formatted SQL file to check s that have labels, but those labels do not match the check configuration.

does not allow the SqlUserDefinedPatternCheck check to be directly enabled or customized by executing liquibase checks enable --check-name=SqlUserDefinedPatternCheck or liquibase checks customize --check-name=SqlUserDefinedPatternCheck because the SearchString does not have a default value. To enable SqlUserDefinedPatternCheck, you need to run the copy subcommand and customize the check:

liquibase checks copy --check-name=SqlUserDefinedPatternCheck

See the Configurable attributes table for the s you can customize.

Then execute the check by running:

liquibase checks run
Restrictions

You cannot delete or reset the SqlUserDefinedPatternCheck check if it is an original check, not a copy of the check.

You cannot reset a customized copy of the SqlUserDefinedPatternCheck check for the initial value because the check does not have a default value for the SearchString . If you no longer need to run this check, delete or disable it.
Enabled by default No

Configurable s

Name Type Description Validation Default value
SearchString String The substring or regular expression to match with the one in the file SearchString should be a valid string or regular expression. There is no default value
Message String The message you want to print when the check detects a pattern match There is no validation The default value is the following: A match for regular expression <SearchString value> was detected in <changeset id>

Warn when tables have more than defined number of columns (check table column count)

Value
Short name TableColumnLimit
Description

The check ensures that no table has more than a threshold number of columns.

Uses

You may have a formatted SQL file or a JSON, YAML, or XML file with one or more s that contain a table with N columns in which N > MAX_COLUMNS. When you execute liquibase checks run against that , you will receive an alert message about the with the exceeded limit.

does not allow the TableColumnLimit check to be directly enabled or customized by executing liquibase checks enable --check-name=TableColumnLimit or liquibase checks customize --check-name=TableColumnLimit. To enable TableColumnLimit, you need to run the copy subcommand and customize the check:

liquibase checks copy --check-name=TableColumnLimit

See the Configurable attributes table for the s you can customize.

Then execute the check by running:

liquibase checks run
Restrictions

You cannot delete or reset the TableColumnLimit check if it is an original check, not a copy of the check.

You cannot reset a customized copy of the TableColumnLimit check for the initial value. If you no longer need to run this check, delete or disable it.
Enabled by default Yes

Configurable s

Name Type Description Validation Default value
MAX_COLUMNS INT The substring or regular expression to match There is no validation The default value is 50

checks settings file

Another part of the quality checks configuration is the checks settings file.

The checks settings file is the file that includes the configuration of checks for a specific project. The file is encoded and should not be manually edited. Use the checks subcommands to copy, customize, enable, or disable rules and update the file.

You can share the checks settings file in your source code control or artifact repository for versioning and consistent use across teams and automation.

Creating the checks settings file

When you start using the checks functionality by running any checks subcommand, will create a checks settings file in the working directory for you –liquibase.check-settings.conf.
