Contexts

Contexts are expressions that control whether commands like update run certain s. You add contexts directly to s in your and specify them at runtime using the --contexts  in the CLI. Context names are case-insensitive strings.

contextFilter rules

By default, a database update runs all s in the , regardless of what you specify in the CLI. If you add a contextFilter to a , it only runs when you specify that context in the CLI, but unmarked s still run. If you do not specify any contexts in the CLI at runtime, every in your runs, even if they have contextFilters attached. Learn more about the contextFilter logic at this blog post.

contextFilter use case examples:

If you want to tag your environment and let the changesets contain the filter logic, set --contexts on the CLI and contextFilter in the changeSet.

If you want to tag your changesets and control the filter logic in the CLI, set the contextFilter on the changeSet and --contextFilter on the CLI.

Examples of SQL, XML, JSON, and YAML s using the contextFilter :

SQL 
--changeset bob:1 contextFilter:test
insert into news (id, title) values (1, 'Liquibase 0.8 Released')
 XML 
<changeSet id="2" author="bob" contextFilter="test">
        <insert tableName="news">
            <column name="id" value="1"/>
            <column name="title" value="Liquibase 0.8 Released"/>
        </insert>
</changeSet>
 JSON 
{
    "changeSet": {
        "id": "2",
        "author": "bob",
        "contextFilter": "test",
        "changes": [
            {
                "insert": {
                    "tableName": "news",
                    "columns": [
                        {
                            "column": {
                                "name": "id",
                                "value": "1"
                            }
                        },
                        {
                            "column": {
                                "name": "title",
                                "value": "Liquibase 0.8 Released"
                            }
                        }
                    ]
                }
            }
        ]
    }
}
 YAML 
- changeSet:
    id: 2
    author: bob
    contextFilter: test
    changes:
      - insert:
          tableName: news
          columns:
            - column:
                name: id
                value: 1
            - column:
                name: title
                value: "Liquibase 0.8 Released"

Context syntax

You can specify a context using AND, OR, !, and parentheses in the s. Without parentheses, the order of operations is !, AND, and then OR.

Examples:

  • contextFilter="!test"
  • contextFilter="v1.0 or map"
  • contextFilter="!qa and !main"

Using a "," to separate contexts works like an OR operation but with the highest precedence.

Examples:

  • "test, qa" is the same as "test OR qa"
  • "test, qa and main" is the same as "(test) OR (qa and main)"

Availability:

  • "," separator is available in all versions of
  • "AND, OR, !, parentheses" are added in 3.2.0
  • Prior to 4.13.0, the "contextFilter" attribute was called "context". A "context" attribute continues to act as an alias for "contextFilter"

Running contexts

You can only specify context filtering logic in a definition, but you can still specify multiple contexts when running in the CLI. However, you can only list out all the contexts that apply to the current run.

liquibase --changelog-file=changelog.xml --contexts="test" update

If your includes several s with complex and simple context filters such as contextFilter="qa and main and !dev" for 1 and contextFilter="test" for 2, you need to pass the following on the command line to deploy them:

liquibase --changelog-file=changelog.xml --contexts="test,qa,main" update

Using contexts for test data

If you manage your test data with , it is best practice to have this data in line with all your other s, but marked with a "test" contextFilter. When you want your test data inserted, run a database update and specify the "test" context in the CLI. When you need to migrate your production database, don't include the "test" context, and your test data will not be included.

If you do not specify any contexts in the CLI at runtime, every will be applied, including those marked with a "test" context.

If you have multiple test environments or test data sets, simply tag them with different contexts, like "min-test" and "integration-test".

Using contexts to control test data is better than having a separate s tree because later s and changes will be applied to existing test data the same as they are applied to production data. If you had a set of test data that was created and simply added after the database is set up, you would be constantly manually updating your test data scripts to keep them in line with the current database schema.

Using contexts for multi-DBMS s

You can use contexts to control which s are run on which databases, but there is an option to use the built in "dbms" tag on the tag.

Default context

You can specify a contextFilter in the root node to assign that context to all s in the by default.

The specified contextFilter will have AND with any contextFilters specified in s within the file.

include and includeAll contexts

You can specify a contextFilter in <include> or <includeAll> tags. If specified, the given contextFilter is added to all s in the included file(s).
