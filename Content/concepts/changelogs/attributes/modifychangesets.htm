modifyChangeSets

The modifyChangeSets tag lets you include SQL scripts in your   while using an application's native executor—like PSQL, SQL Plus, or SQLCMD—instead of the JDBC default. You can use it to specify an overarching executor for all files in your project.

Uses

If you have many s that need to be run with a native executor, and you organize your s in a "root" and "nested" format using the include and includeAll tags, you can use the runWith  to specify an executor that applies to everything in modifyChangeSets. Since your nested s inherit the executor specified in modifyChangeSets, you don't have to specify it on each individual you include.

If you want to make an exception for a particular you're including, you can specify the default JDBC executor for that . All other s still run with the executor inherited from modifyChangeSets.

You can still include additional files in your root outside of the modifyChangeSets tag. These files run with the JDBC executor.

Syntax

s

Value Notes
runWith jdbc Default value if none specified
psql Executor for PostgreSQL
sqlplus Executor for MSSQL Server
sqlcmd Executor for Oracle
XML example 


    <modifyChangeSets runWith="sqlplus">
        <include file="my/nested/changelog.xml" />
        <includeAll path="release200/changelogs/sql" />
    </modifyChangeSets>

    <include file="sql/another.sql"/>
</databaseChangeLog>
 YAML example 
databaseChangeLog:
  - modifyChangeSets:
      runWith: sqlplus
      include:
        - file: my/nested/changelog.xml
      includeAll:
        - path: release200/changelogs/sql
  - include:
      file: sql/another.sql
 JSON example 
{
   "databaseChangeLog": [
       {
           "modifyChangeSets": {
               "runWith": "sqlplus",
               "include": {"file": "my/nested/changelog.xml"},
               "includeAll": {"path": "release200/changelogs/sql"}
           },
           "include": {"file": "sql/another.sql"}
       }
   ]
}
