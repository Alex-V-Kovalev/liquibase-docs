Design Your Project

Your is the core of your project. While you can use one for all your deployments, this may not scale well. Instead, you can use multiple s for different purposes. To organize multiple s, you can use the include and includeAll tags to create a main (root/parent) and one or more nested (child) s. This document explains some best practices on structuring your s.

Choose a schema design pattern

You can use in two ways:

  • Shared schema: All developers work in a single, shared database schema
  • Multi-schema: Each team works in a different database schema

To choose what's best for your organization, consider the processes your developers currently use to deploy changes to the database.

Shared schema

If you use a shared schema with , all your developers work on the same database schema.

Using a single schema can make it simple to design, manage, and query your database. You may also find it easier to keep your data consistent and error-free if you use a single schema. However, using a single schema in a large database may become confusing as you create more objects. If you need to write long queries to navigate complex object structures in a single-schema database, the queries may negatively affect performance.

Also, using a shared schema requires additional coordination between teams to ensure they don't overwrite or break each other's work. For example, if one team makes a change to a script that is a dependency of another team's automated system, their workflow may break. To solve this, you must agree on a development process that minimizes conflicts. One solution is to keep all schema content in a single repository owned by one team. When other teams want to make changes, they submit pull requests from their source control tool. Alternatively, each team can work in their own directory in the schema and try to minimize cross-team dependencies.

Multiple schemas

If you use multiple schemas with , you create one or more schemas in your database for different teams.

Using multiple schemas may make it easier to organize your data: you can structure and customize the data in each schema individually. This may be helpful in large databases containing many unique data structures. Also, using multiple schemas may prevent conflicts with source control merges or broken dependencies and may help you store data backups. Using multiple schemas can also improve security by allowing you to limit permissions for different users and applications in each schema.

However, using multiple schemas means you may have to duplicate some of your data, which can make consistency more difficult. You may also need to call multiple times to send your code to production. Depending on your automation tooling and workflow, this design may require extra maintenance from your DevOps team, but you may be able to deploy across all schemas from a single connection with a superuser account.

Choose a structure

When designing your project, it is a best practice to create a simple main that you don't have to modify directly. Instead, you can make changes to each of the s beneath it, which you can reference with the include and includeAll tags. The nested s can be modified and deployed independently, but you can also deploy all changes in a single update from the main . There are two common design patterns, object-oriented and release-oriented. The design you choose should correspond to the structure of your existing code repository and workflow.

Object-oriented

For most organizations, it is a best practice to use an object-oriented design pattern.

In an object-oriented structure, you create one per object or object type. For example:

com
  example
    db
      changelog
        changelog-main.xml
        changelog-index.xml
        changelog-procedure.xml
        changelog-table.xml
        changelog-view.xml

In this example, we have a main called changelog-main.xml which contains include tags referencing changelog-index.xml, changelog-procedure.xml, changelog-table.xml, and changelog-view.xml.

Alternatively, you can create directories for each type of object, each containing one or more s per particular database object:

com
  example
    db
      changelog
        changelog-main.xml
        changelog-indexes
          my-favorite-index.xml
          that-other-index.xml
        changelog-tables
          employees.xml
          customers.xml

In this example, we have a main called changelog-main.xml which contains includeAll tags referencing the directories /changelog-indexes and /changelog-tables. The s you modify regularly are contained in the nested directories rather than in the main /changelog directory.

These designs are useful because they allow you to easily see changes made over time to specific objects (like a particular table) or groups of objects (like all tables in your database). An object-oriented format makes it easier to roll back changes to particular objects without worrying about unrelated changes in the .

Having a more detailed structure can also potentially simplify the Source Control Management (SCM) process. If all your developers work in the same , they may experience source control merge conflicts more frequently. By separating your s into use-cases for different objects, developers are likely to be working in different files and therefore will not overwrite each other's code.

Release-oriented

If a release-oriented structure, you create one per release or release group. For example:

com
  example
    db
      changelog
        changelog-main.xml
        changelog-1.0.xml
        changelog-1.1.xml
        changelog-2.0.xml

In this example, we have a main called changelog-main.xml which contains include tags referencing changelog-1.0.xml, changelog-1.1.xml, and changelog-2.0.xml.

Alternatively, you can create directories for each major release (or other release group), each containing one or more s per minor release:

com
  example
    db
      changelog
        changelog-main.xml
        changelog-1.x
          changelog-1.0.xml
          changelog-1.1.xml
        changelog-2.x
          changelog-2.0.xml

In this example, we have a main called changelog-main.xml which contains includeAll tags referencing the directories /changelog-1.x and /changelog-2.x. The s you modify for a particular release are contained in the nested directories rather than in the main /changelog directory.

A release-oriented structure lets you bundle all content associated with a release in one place. However, this design can be difficult to maintain. If you want to see changes made to a particular object over time, the release-oriented structure requires you to look in multiple locations for that information.

Additionally, having a single per release (even minor release) may lead to more frequent source control merge conflicts. You can solve this by creating multiple s within each release directory, such as /changelog-1.0/indexes.xml and /changelog-1.0/tables.xml. However, this structure may be more complicated to navigate and maintain than a pure object-oriented design.

Managing your s

Preconditions

You can specify preconditions in your s and individual s to define security and standardization requirements for your s. By default, if a global precondition (on the level) fails, halts the deployment. If a local precondition (on an individual ) fails, does not deploy that , but continues with the rest of the deployment. You can configure this behavior according to your needs.

s

s are pieces of metadata in the header of your or individual s.

If you want to run specific s in your , you can mark them with  filtering logic called contexts and labels. For example, you can use contexts to group together changes in different environments (like dev and prod) and labels to mark s as belonging to a particular feature or version. At runtime, when you run a command like update, you can specify those contexts or labels in the CLI. To decide whether to use contexts, labels, or both, consider the following:

  • If any complex logic is being handled by the author, you may wish to use contexts.
  • If any complex logic will be determined at runtime, you probably want labels.

You can also specify that certain s run on every deployment (runAlways) or only when modified (runOnChange), and more.
