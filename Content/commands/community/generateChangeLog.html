generateChangeLog command


The generateChangeLog command creates a file that has a sequence of s which describe how to re-create the current state of the database.

Uses

The generateChangeLog command is typically used when you want to capture the current state of a database, then apply those changes to any number of databases. This is typically only done when a project has an existing database, but hasn't used before. See Adding [%=General.Liquibase%] on an existing project for more details.

When using the update command to apply the changes in the , will not create a new database or schema. You must create them before applying the to it.

Running the generateChangeLog command

To generate a :

  1. Configure the file to include your driver, classpath, and URL for the database you want to capture.

For more information on how to configure your file, see the Creating and configuring a [%=General.liquiPropFile%] file topic. Instead of using a file, you can also pass the necessary information on the command line.

  1. Open your CLI and run the following command:
liquibase --changeLogFile=dbchangelog.xml generateChangeLog

If you want to create an SQL file, add your database type name when specifying the file: liquibase --changeLogFile=mychangelog.oracle.sql generateChangeLog. Replace .oracle.sql with your database type. When in doubt about your database type name, check Supported Databases.

generateChangeLog global s

 Optional
--includeCatalog=[boolean] Includes the catalog in a generated s if the value is true. The default value is false. Optional
--includeSchema=[boolean] Includes the schema in a generated s if the value is true. The default value is false. Optional
--includeTablespace=[boolean] Includes the tablespace of tables and indexes in a generated s if the value is true. The default value is false. Optional *
--dataOutputDirectory=DIR Sends the data output as a CSV file in the given directory. Optional
--diffTypes Includes a list of diff types in a file expressed as a comma-separated list (without spaces) from: catalog,tables,functions,views,columns,indexes,foreignkeys,primarykeys,uniqueconstraints,data,storedprocedure,triggers,sequences. Optional **

* --includeTablespace only captures the tablespace if it was specified in the create table statement.

** If the --diffTypes value is null, then the default types will be: tables, views, columns, indexes, foreignkeys, primarykeys, uniqueconstraints.

Output

The generateChangeLog command generates a that contains all your Objects (represented as s) and places the file in the same directory where the command was ran.

The extension you provide determines the format of the , so if you specify the filename as changelog.xml, you will get an XML formatted . However, if you specify the filename as changelog.yaml, changelog.json, or changelog.postgresql.sql, you will get s formatted in YAML, JSON, or SQL respectively.

Additional functionality with

While stores all s in a , creates a directory called Objects and places the directory at the same level as your . The Objects directory contains a subdirectory for each of the following stored logic types:

  • package
  • packagebody
  • function
  • stored procedure
  • trigger
  • view

Not all database platforms support all of these types.
